Before Implementing the Frankly SDK

  1. You’ll need to obtain an appKey and appSecret before implementing the Frankly SDK. These are required for authenticating with our server. If you don’t have these, please contact us at platform-support@franklychat.com.
  2. Your server will need to be able to generate a JSON Web Token (JWT). To learn more about how to generate a JWT, click here.

Basic Integration of the Frankly SDK

In your AppDelegate, import the Frankly SDK.

1
#import "FranklySDK.h"

Then initialize the SDK in application:didFinishLaunchingWithOptions:

1
[FranklySDK initializeSDKWithServerType: FRSDKServerTypeProduction];

The Frankly SDK provides a chat experience within your existing app. Users are generally presented with a list of rooms. Within the rooms is where users communicate with each other, and with your brand.

All users require some level of authentication before being admitted into the Frankly chat experience. You will decide whether your users can enter rooms anonymously, as a guest (with a one-time username), or as a logged-in user. This designation is made via the ‘role’ parameter in initiateAuthenticationWithUserID:role: (see code block below).

Users that enter the room anonymously will have read-only access, and will be offered a way to signup or login. If they choose to signup/login, you can customize whether they see the option to enter as a guest, as a logged-in user, or whether they see options for both. This is customized via the Frankly SDK’s loginBehavior property (see code block below).

Specify your desired login behavior and set a delegate for the authentication callbacks. And then kick off authentication, passing in the user’s ID and role. Note that the users ID here is NOT a Frankly ID. It is the ID that your application uses for identifying that unique user.

1
2
3
    [FranklySDK sharedInstance].loginBehavior = FRSDKLoginBehaviorPartner;
    [FranklySDK sharedInstance].authenticationDelegate = self;
    [[FranklySDK sharedInstance] initiateAuthenticationWithUserID:@"MyUserID123" role:FRSDKAuthenticationRoleUser];

In the code above, you assigned 'self’ as the delegate to the authentication callbacks. Indicate that this class conforms to the FranklySDKAuthenticationDelegate.

1
    @interface ViewController : UIViewController <FranklySDKAuthenticationDelegate>

And then implement the delegate callback methods for authentication. Note: this requires that your organization have an endpoint for creating a JSON Web Token.

Set up a Room List

The room list is displayed in a navigation controller. When a user selects a room, a room view controller is pushed onto the stack. You can either supply your own navigation controller, or use the one provided by the Frankly SDK, which has the Frankly navbar.

Using your own navigation controller.

1
2
    UIViewController *viewController = [[FranklySDK sharedInstance] indexViewController];
    UINavigationController *navController = [[UINavigationController alloc] initWithRootViewController:viewController];

Or using Frankly’s navigation controller, which has the Frankly navbar.

1
    UINavigationController *navController = [[FranklySDK sharedInstance] indexNavigationController];

Setting User Info

In order to have the user’s name and profile image associated with their chat activity, the Frankly SDK must be given this information. Otherwise, the user’s activity will appear under the anonymous title “User”, which doesn’t provide for the best experience if the user has entered chat as a logged in user.

The user’s name and profile image should be set in the authentication delegate’s authenticationDidSucceedWithUserID: method.

1
2
3
4
    - (void)authenticationDidSucceedWithUserID:(NSNumber *)userID {

        [[FranklySDK sharedInstance] setUserName:@"Ezra Hamilton" profileImage:nil];
    }

You can customize whether the user can edit their name and image within the chat experience.

1
2
3
    - (BOOL)shouldAllowProfileEditing {
        return YES;
    }

The above method requires conformance to the FranklySDKUserProfileDelegate protocol. You will also need to assign the delegate that will implement shouldAllowProfileEditing and the other delegate methods associated with the protocol.

1
2
3
4
5
6
7
8
    @interface ViewController : UIViewController <FranklySDKAuthenticationDelegate, FranklySDKUserProfileDelegate>

    - (void)viewDidLoad {
        [super viewDidLoad];
        [FranklySDK sharedInstance].userProfileDelegate = self;

        ...
    }

The profile delegate will get called every time the user profile image or display name changes, whether it’s set programmatically or via the user interface.

1
2
3
4
5
6
7
    - (void)user:(NSNumber *)userID didUpdateDisplayName:(NSString *)displayName {
        NSLog(@"userID: %@, did update displayName: %@", userID, displayName);
    }

    - (void)user:(NSNumber *)userID didUpdateProfileImage:(UIImage *)image {
        NSLog(@"didUpdateUserImage");
    }

Customization

FranklySDK on iOS allows partners to set values that affect FranklySDK’s interface within the partner app. This data is passed in via a standard plist file. An example data file, FRSDKExampleConfig.plist, is included FranklySDK package.

Partners may modify that file, or create a new one and then, any time after initializeSDKWithServerType: has been called,add the file to FranklySDK via the function addCustomizationFile:(NSURL *)fileURL. Multiple files may be added via this call, any collisions with existing keys will cause the existing value to be overwritten by the new value.

Customization Plist

This is a standard Apple plist file which means it’s an XML file with a single object named Root. FranklySDK expects this Root object to be a dictionary of dictionaries. Each dictionary represents a customization category that is populated with key value pairs.

Data Types

The base values stored in the dictionaries in the customization file will be of the following types:

Data Types This is a string that follows a standard hex string format with alpha. It maybe be preceded by ‘0x’ or ‘#’ but the last 8 characters should represent the hex values in red, green, blue, alpha order. So the color white with full opacity would be represented as 0xFFFFFFFF and black, half opacity would be represented as 0x0000007F.

Customization Categories

the ui key contains a dictionary of key value pairs with the following keys:

Color Key Use Case
primary-fg-color Takes a color value and is used throughout the app as primary foreground color.
primary-bg-color Takes a color value and is used throughout the app as the primary background color.
accent-bg-color Takes a color value and is used throughout the app as an accent color. Most noticeably as the background of the locked Input Bar.
brand-bg-color Takes a color value. Is used as the background for Sticky Messages when they are docked at the top of a Room.
attention-bg-color Takes a color value. Is used in the custom activity indicators.
