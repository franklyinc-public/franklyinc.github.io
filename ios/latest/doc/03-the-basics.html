Setting Up Your Room List

Important Note Please make sure you’ve set up at least one Room in the Admin Console before proceeding. If you’re unsure of how to do this or if you don’t have Admin Console access, please see Getting Started for next steps.

Calling the Room List via the iOS SDK

To call the Room List, retrieve the room list controller and present it in your app or push it on your navigation controller. You have two options for room list controller to call, depending on your app’s configuration: - [FranklySDK indexViewController] - Retrieves a view controller that returns only the Room List. - [FranklySDK indexNavigationController] - Retrieves a view controller that returns the Room List with the Frankly navbar.

For partners wanting to use Conditional Rooms, calling either indexViewController or indexNavigationController will present the user with Conditional Rooms you have already configured in the Frankly Admin Console.

Note [FranklySDK roomListViewController] and [FranklySDK roomListNavigationController] are being deprecated by the end of September, 2015.

Setting the Display Name & Profile Avatar

A couple things to note about these setting these properties for a user’s Frankly profile:

  • The maximum length of a Frankly Display Name is 18 characters
  • For best results, we recommend uploaded Profile Avatars to be 250x250px compressed JPGs
  • Important Note The Frankly Chat Platform automatically provides default Profile Avatars for users where it has not been set for them, to provide a nicer out-of-the-box experience

The profile delegate will get called every time the user profile image or display name changes. Also, you can enable or disable user editing of their profile data within the FranklySDK UI by allowing your delegate respond appropriately to the -(BOOL)shouldAllowProfileEditing delegate request.

Setting the user’s UserName and profileImage can be done via the calls through the FranklySDK object:

1
-(void)setUserName:(NSString *)userName profileImage:(UIImage *)image];

When the user’s UserName and profileImage have been updated and synced with Frankly servers, the profileDelegate will get the callbacks didUpdateDisplayName and didUpdateProfileImage, confirming that these properties have been set. At the moment, the profileDelegate does not return a failure callback, so please handle accordingly. Please also keep in mind, that in the event that you want to set UserName or profileImage, you can pass in nil for whichever property you do not wish to update. Also note that the call to set these properties does nothing if called before receiving authenticationSucceeded delegate message.

Customizing UI in the SDK

This section will describe how the Frankly SDKs allow you to customize various components of the provided UI.

Customizing the iOS SDK

FranklySDK on iOS allows partners to set values that affect FranklySDK’s interface within the partner app. This data is passed in via a standard plist file. An example data file, FRSDKExampleConfig.plist, is included FranklySDK package.

Partners may modify that file, or create a new one and then, any time after initializeSDKWithServerType: has been called,add the file to FranklySDK via the function addCustomizationFile:(NSURL *)fileURL. Multiple files may be added via this call, any collisions with existing keys will cause the existing value to be overwritten by the new value.

Customization Plist

This is a standard Apple plist file which means it’s an XML file with a single object named Root. FranklySDK expects this Root object to be a dictionary of dictionaries. Each dictionary represents a customization category that is populated with key value pairs.

Data Types

The base values stored in the dictionaries in the customization file will be of the following types:

  • Data Types This is a string that follows a standard hex string format with alpha. It maybe be preceded by ‘0x’ or ‘#’ but the last 8 characters should represent the hex values in red, green, blue, alpha order. So the color white with full opacity would be represented as 0xFFFFFFFF and black, haf opacity would be represented as 0x0000007F.
Customization Categories
  • ui ** the ui key contains a dictionary of key value pairs with the following keys:
Color Key Use Case
`primary-fg-color` Takes a color value and is used throughout the app as primary foreground color.
`primary-bg-color` Takes a color value and is used throughout the app as the primary background color.
`accent-bg-color` Takes a color value and is used throughout the app as an accent color. Most noticeably as the background of the locked Input Bar.
`brand-bg-color` Takes a color value. Is used as the background for Sticky Messages when they are docked at the top of a Room.
`attention-bg-color` Takes a color value. Is used in the custom activity indicators.
